To explain dependencies from my understanding I would say they are a specific code that developer use in their projects and the project will not run if the code is not there. We can get these codes as packages from other developers who shared their codes on coding platforms, for examaple, GitHub or NPM, downloading the zip file with JavaScript code and other files, and adding it to your project is what is called a dependency. According to Doglio, there are five different types of dependencies (2020). Web developers use these dependencies to help support their code and not let it depend on the details of the actual dependencies they are using which might require them to change their code unnecessarily.

These different types of dependencies are, runtime (normal) dependencies, peer dependencies, development (Dev) dependencies, bundled dependencies, and optional dependencies (Doglio, 2020). I will briefly explain each dependency, before I jump into explaining the one, I used in my project and how it has strengthened the user experience and why I found it suitable for the web page I used it on.

Runtime dependencies are used to specify names for keys and values, then enable the package manager they use to take care of grabbing the values and names from the global registry known as npmjs.org (Doglio, 2020). This dependency is considered one of the easiest because they use values (version) and file name for file transfers and storages to keep the project functioning. The developer can use URLs to download versions that they see suitable for their projects as well as local files whereby they will be able to connect directly to their files to test and run their codes. These normal dependencies do not require the developer to rely on other libraries or APIs for their projects to work since they have everything the project needs to function (Doglio, 2020).

Peer dependencies on the other hand, are responsible for determining the packages a developer’s project is cooperative with. These are specifically used to specify a delicate relationship with another project, a developer can feel like their projects need a certain dependency but not precisely use it or the dependency is already installed by some other project, this then can be seen as a peer dependency. Doglio uses a couple of examples for when the developers can use this dependency, such as when you are establishing a Micro Frontend, and trying to figure out which dependencies are external, and which are internal (2020).

Dev dependencies store modules the developer uses only during the development stage of their project. It is basically seen when the developer can deploy to production without the use of a package, they work only if one installs them through npm link or install from the root folder of their project (Doglio, 2020). Developers should then give thought to using these dependencies only if the modules and packages are going to be used by additional projects.

Furthermore, we have bundled dependencies, which are used when packing a project into a unique file. The process is done through npm pack order that develops the folder into a zip file. Lastly, optional dependencies are more similar to normal dependencies, but they are different in a way that NPM will not be pulling out with an error if it cannot install the packages (Doglio, 2020). Instead of running an error when a certain link or version is not found, the NPM allows the project to keep running, and all this can be implemented through code. This dependency can be used when a developer depends on an unreliable source in their projects. This topic was really interesting and eye-opening because I had no idea I had so many options and dependencies to use in my projects, the only one I knew a bit about was the dev dependency. Dependency is simply the process of installing packages or copy and pasting other people’s codes from GitHub for example to make your project depend on it, which means the project will not function without the code.

For my project I used the normal dependencies and went to GitHub to download local files that a developer called davidfig cared to share on the 15th of August 2020, for the click events in JavaScript into my project to help ease the flow and responsiveness in my portfolio page. I chose this local file because I believe it is suitable for the type of structure and user experience I was aiming for, which is enabling the user to navigate through the images in my portfolio page effortlessly and get an effective response when they click on buttons to view the different images they would like to look at as well as the links available on the webpage.
